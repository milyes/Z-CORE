Kanidm — Configuration SSO Souverain

Identity Provider · OIDC · WebAuthn · MFA
Kanidm v1.9.x Debian 12 Docker Compose Caddy 2 Mars 2026

1. docker-compose.yml

services:
  kanidm:
    image: kanidm/server:latest
    container_name: kanidm
    restart: unless-stopped
    volumes:
      - kanidm_data:/data
    networks:
      - sovereign_net
    healthcheck:
      test: ["CMD", "kanidmd", "healthcheck", "--config", "/data/server.toml"]
      interval: 30s
      timeout: 10s
      retries: 3

volumes:
  kanidm_data:
    driver: local

2. server.toml

version = "2"

bindaddress = "[::]:8443"

domain = "idm.tundomaine.fr"
origin = "https://idm.tundomaine.fr"

trust_x_forward_for = true

db_path = "/data/kanidm.db"
log_level = "info"

[online_backup]
path     = "/data/backups/"
schedule = "0 3 * * *"  # backup quotidien 3h
Important : origin doit correspondre exactement à l'URL publique vue par le navigateur via Caddy. Toute divergence bloque l'authentification OIDC.

3. Configuration Caddy

idm.tundomaine.fr {
    reverse_proxy kanidm:8443 {
        header_up X-Forwarded-For   {remote_host}
        header_up X-Forwarded-Proto {scheme}
        header_up Host              {http.request.host}
    }
}

4. Initialisation (première fois)

# Démarrer le conteneur
docker compose up -d kanidm

# Récupérer les comptes admin (affiche les mots de passe initiaux)
docker exec -it kanidm kanidmd recover-account admin
docker exec -it kanidm kanidmd recover-account idm_admin

# Vérifier le statut
docker exec -it kanidm kanidmd healthcheck --config /data/server.toml

5. Clients OIDC — Création par service

Service Commande kanidm Notes
Nextcloud kanidm system oauth2 create nextcloud "Nextcloud" https://cloud.tundomaine.fr App OIDC native
Matrix Dendrite kanidm system oauth2 create matrix "Matrix" https://chat.tundomaine.fr homeserver.yaml
Stalwart Mail LDAP en mode backend ou Token Introspection LDAP fallback
Vaultwarden kanidm system oauth2 create vaultwarden "Vault" https://pw.tundomaine.fr SSO_ENABLED=true
Jitsi Meet kanidm system oauth2 create jitsi "Jitsi" https://meet.tundomaine.fr JWT auth
Collabora kanidm system oauth2 create collabora "Collabora" https://office.tundomaine.fr Nextcloud intégré

6. Groupes par service

# Créer les groupes d'accès
kanidm group create nextcloud_users
kanidm group create matrix_users
kanidm group create stalwart_users
kanidm group create vaultwarden_users
kanidm group create jitsi_users
kanidm group create admins_sudo

# Ajouter un utilisateur à un groupe
kanidm group add-members nextcloud_users alice
kanidm group add-members admins_sudo bob

# Lier groupe au client OAuth2
kanidm system oauth2 update-scope-map nextcloud nextcloud_users email profile openid

7. Forcer WebAuthn + MFA

# Politique MFA obligatoire pour tous
kanidm system policy account-policy set \
  --mfa-require true \
  --webauthn-attest-no-phishing true

# Forcer WebAuthn sur un compte spécifique
kanidm person credential update alice \
  --add-webauthn "yubikey"

8. Bonnes pratiques

  • Forcer WebAuthn + MFA pour tous les utilisateurs dès l'installation
  • Créer un groupe par service — jamais de groupe fourre-tout "all_users"
  • Intégrer les backups /data/backups/ dans Restic vers NAS chiffré
  • Monitorer via Prometheus — Kanidm expose des métriques natives
  • Avant mise à jour : docker compose pull → backup → up -d
  • Ne jamais exposer le port 8443 directement — toujours passer par Caddy
  • Logs dans Loki/Grafana pour audit des authentifications
✅ Prêt pour la production : Cette configuration est celle utilisée dans la stack Z-CORE sur Hetzner CPX31, Debian 12, avec Caddy 2 comme reverse proxy et Ansible pour le provisioning idempotent.

Intégration ZTA ↔ Kanidm

Comment Kanidm implémente chaque pilier ZTA

Kanidm est le cœur du Zero Trust Architecture dans Z-CORE. Voici comment il couvre concrètement chacun des 5 piliers ZTA.

Pilier 1 — Identity Trust → Kanidm SSO

Exigence ZTAImplémentation Kanidm
Identité vérifiée à chaque accèsToken OIDC court-vivant (1h), refresh via session Kanidm
MFA obligatoireTOTP + WebAuthn natif, policy enforced
Comptes de service isolésService Accounts avec secrets rotatifs automatiquement
Fédération d'identitéOIDC pour Nextcloud/Matrix/Vault, LDAP pour Stalwart
Audit de toutes les authentificationsLogs structurés → Loki → Grafana

Pilier 2 — Device Trust → Certificats Kanidm

Exigence ZTAImplémentation
Enregistrement appareilCertificat client émis par Kanidm CA interne
Health check appareilAttributs de session conditionnels (appareil vérifié)
Révocation immédiatekanidm device revoke <id> — effectif instantanément

Pilier 3 — Workload → PDP + PEP via Kanidm

# Flux d'authentification Zero Trust complet

1. Client → Caddy (TLS 1.3)
2. Caddy → Kanidm (OIDC authorize endpoint)
3. Kanidm → MFA challenge (WebAuthn / TOTP)
4. Kanidm → Token JWT signé (claims: groups, device, mfa_level)
5. Caddy → Service (avec token dans header Authorization)
6. Service → Kanidm token introspection (validation)
7. Service → Accès accordé ou refusé (PEP enforcement)

Pilier 4 — Data Protection → Chiffrement + Kanidm

  • Kanidm gère les clés de chiffrement pour Vaultwarden (AES-256)
  • Accès PostgreSQL conditionnel au token Kanidm valide (mTLS + RBAC)
  • Matrix E2EE : Kanidm comme key backup pour les clés de session
  • Stalwart : certificats S/MIME liés aux comptes Kanidm

Pilier 5 — Micro-segmentation → Groupes Kanidm

# Chaque segment réseau = un groupe Kanidm
# Seuls les membres du groupe accèdent au service correspondant

kanidm system oauth2 update-scope-map nextcloud  nextcloud_users  openid email profile
kanidm system oauth2 update-scope-map matrix     matrix_users     openid email profile
kanidm system oauth2 update-scope-map stalwart   stalwart_users   openid email
kanidm system oauth2 update-scope-map vaultwarden vault_users     openid email
kanidm system oauth2 update-scope-map jitsi      jitsi_users      openid email

# Un utilisateur sans groupe n'a accès à rien — Zero Trust par défaut
